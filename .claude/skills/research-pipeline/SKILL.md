# research-pipeline — organize, build, critique, approve

Every substantive request becomes a **project folder in the vault** with a fixed shape. Goal:
each project tells its story on its own (diagram + figure), proves it with calculation (script +
notebook), and is sharpened by two independent critics before it's filed as done.

Write into the user's vault (`~/vault/TV`, or via the progress3d MCP `write_file` for cloud agents).
One folder per project: `projects/<kebab-slug>/`.

## 1 · Tracker  → `tracker.md`
Frontmatter (`project, status, created, tags`) + the **user request VERBATIM** (blockquote) + a
marker **done / tbd / ongoing** + a one-line finding + links to the 4 artifacts. This is the index.

## 2 · Minimal report (4 artifacts — minimal, NO redundancy)
- **diagram** (`diagram.html`) — a vivid three.js diagram (house style, see [[render-3d]] skill) that
  conveys the story *on its own*: labels, legend, the key answer called out.
- **script** (`experiment.py`) — the model/experiment as the **single source of truth**: small,
  parameterized, functions only. The notebook imports from it — never duplicate logic.
- **result figure** (`result.png`) — generated by the script; proves the method's value; annotate the
  finding directly on the plot.
- **notebook** (`analysis.ipynb`) — imports the script, narrates assumptions → calculation → figure →
  conclusion. Build with `nbformat` and EXECUTE (`ExecutePreprocessor`) so outputs embed.
- (+ `SOURCES.md` when claims need grounding; web-search the key numbers, cite them, flag estimates.)

**Everything needs calculation** — no hand-waving numbers. Intricate structure ⇒ visualize it.

## 3 · Critique & approval (Kimi + GLM as reviewers AND audience)
Screenshot the diagram (headless Chrome) and run both critics on the four artifacts. Ask exactly:
- (a) is the **diagram** intuitive and self-contained (story clear without the notebook)?
- (b) does the **result figure** prove the method's value / is the finding interesting & promising?
- (c) is the **code** (script + notebook) readable, clear logic, and **FREE OF REDUNDANCY** — no
  duplicated kernels/logic, no dead code? (the user cares most about this.)
Force structured JSON verdicts. Same spawn pattern as `tools/judge/judge.mjs`:
- Kimi: `kimi -p "<brief>"` (reads images via ReadMediaFile; `-p` can't combine with `-y`/`--auto`).
- GLM: `claude -p "<brief>" --permission-mode bypassPermissions` with env
  `ANTHROPIC_BASE_URL=https://api.z.ai/api/anthropic`, `ANTHROPIC_AUTH_TOKEN=$GLM_API_KEY`,
  `ANTHROPIC_API_KEY` unset (see [[progress3d-visual-judge]] / [[progress3d-vault-bridge]]).
Apply the consensus/high-value fixes; **reject bad suggestions with a reason** (don't apply blindly).
Re-render to verify each fix actually landed.

## 4 · Approve  → `progress/`
Copy the revised 4 artifacts into `projects/<slug>/progress/` + a concise **`critique-summary.md`**
(scores, what was applied, what was rejected and why). Flip the tracker marker to `done`. Leave the
vault for the user to push manually (Obsidian Git "Commit-and-sync" — they want no auto-sync latency).

> **Always pull before push.** If you ever commit to the vault yourself, `git pull --no-rebase` first
> (the vault is shared by many agents) so you don't clobber others' work or get a rejected push.

## Reference run
`projects/spacex-ipo-beneficiary/` — "SpaceX IPO benefits which company the most?" Two-winner finding
(Alphabet absolute $, Baron Partners relative %); full 4 artifacts + Kimi/GLM critique round.
