# research-pipeline — the report IS a 3D research map

**Core principle:** a report is **a progress3d 3D map with a note embedded on every node** — NOT a 3D
animation embedded in a note. The map is the container; you fly around it and click a node to read its
note (the summary, a figure, the method, the sources, or a domain entity). Everything is calculated
(no hand-waved numbers); intricate structure ⇒ visualize it as the map's shape.

Write into the user's vault (`~/vault/TV`, or via the progress3d MCP for cloud agents).

## 1 · Raw materials  → `projects/<kebab-slug>/`
The calculation + assets that the map's notes will draw from (minimal, NO redundancy):
- **`experiment.py`** — the model/experiment as the **single source of truth**: small, parameterized,
  functions only. The notebook imports it — never duplicate logic.
- **`result.png`** — generated by the script; proves the method's value; annotate the finding on-plot.
- **`analysis.ipynb`** — imports the script; assumptions → calculation → figure → conclusion. Build with
  `nbformat` and EXECUTE (`ExecutePreprocessor`) so outputs embed.
- **`diagram.html`** — optional rich standalone three.js scene (house style, [[render-3d]] skill) used to
  design the map's layout / as a hero visual. The *navigable* report is the map below, not this file.
- **`SOURCES.md`** when claims need grounding; web-search the numbers, cite them, flag estimates.
- **`tracker.md`** — thin index: frontmatter + the **user request VERBATIM** + marker (done/tbd/ongoing)
  + one-line finding + a link to "Open the 3D map" (the map slug) and to the raw files.

## 2 · The 3D research map  (THE deliverable)
- **Map graph** → `progress3d/maps/<slug>.json`: `{title, nodes[], edges[]}`. Nodes use the **research
  node types** (clean glowing spheres, not transformer cells):
  `hub` (the central subject) · `primary` (the headline/summary) · `doc` (a report section: figure /
  method / sources) · `entity` (a domain entity) · `accent` (the answer / highlighted entity) ·
  `muted` (secondary/inactive). Lay nodes out in 3D so the *shape* carries meaning (e.g. value flows
  hub→entities; report sections orbit the hub). Node `id`s must be globally unique → prefix per project
  (`<slug-abbr>-summary`, `<slug-abbr>-<entity>`).
- **Node notes** → `progress3d/<id>.md` (one per node — this is the note "embedded" in the node; clicking
  the node opens it):
  - `…-summary` (primary): the finding/TL;DR + wiki-links to the other nodes.
  - `…-figure` (doc): `![[result.png]]` (+ `![[diagram.png]]`) with a caption.
  - `…-method` (doc): assumptions table + links to `[[experiment.py]]` and `[[analysis.ipynb]]`.
  - `…-sources` (doc): `[[SOURCES]]` + key caveats.
  - one note per **domain entity** node with its specific numbers.
- **Make it viewable:** drop the json in `progress3d/maps/`; it appears in the plugin's map dropdown.
  Set `settings.activeMap` (plugin `data.json`) to the slug to show it on reload, or let the user pick.

## 3 · Critique & approval — two AGENTIC judges (kimi-code + claude-code/GLM)
Judges are **agents that run/observe**, not one-shot calls. Drive them like `tools/judge/judge.mjs`:
- **kimi** = `kimi -p` (kimi-code agent; headless via the kfc key in `~/.kimi-code/config.toml`).
- **glm** = `claude -p --permission-mode bypassPermissions` with `ANTHROPIC_BASE_URL=https://api.z.ai/api/anthropic`,
  `ANTHROPIC_AUTH_TOKEN=$GLM_API_KEY`, `ANTHROPIC_API_KEY` unset.
Ask each (structured JSON verdict): (a) does the **map** tell the story on its own — clear nodes, sensible
shape, the answer obvious? (b) does the **result figure** prove the method's value / is it interesting?
(c) is the **code** (script + notebook) readable with clear logic and **FREE OF REDUNDANCY** (no duplicated
logic/cells, no dead code — the user cares most about this)? Apply consensus fixes; **reject bad ones with
a reason**; re-render the map (screenshot) to verify fixes landed.

## 4 · Approve
Flip the tracker marker to `done` and write a concise `critique-summary.md` (scores, applied, rejected+why)
as a `…-critique` node note or in `projects/<slug>/progress/`.

> **Always pull before push.** If you commit to the vault, `git pull --no-rebase` FIRST (the vault is
> shared by many agents) so you don't clobber others' work or get a rejected push. Default to leaving the
> push to the user (Obsidian Git "Commit-and-sync") unless they ask.

funds, doc nodes for summary/figure/method/sources. Click a node → its note opens.
